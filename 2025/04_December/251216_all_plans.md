# 일정표에 대한 고민과 UX/사용성 극대화

일정표를 단순한 데이터 나열이 아닌 팀과 개인이 현재 상황을 빠르게 판단하고 다음 행동을 결정하도록 돕는 UI로 재정의했다.  
기능을 더하는 방향이 아니라 덜 보여주되 더 명확하게 전달하는 구조가 필요하다는 결론에 도달했다.  

## 1. 일정표는 입력 폼이 아니라 판단 도구다

기존의 일정 관리 UI는 대부분 두 가지로 나뉜다.

- 입력 중심: 폼은 풍부하지만 한눈에 상황 파악이 어렵다
- 시각화 중심: 보기엔 좋지만 수정과 확장이 어렵다

이번 고민의 핵심 질문은 이것이었다.

이 화면을 열었을 때  
사용자는 무엇을 가장 먼저 판단해야 하는가?

그 답은 명확했다.

- 지금 진행 중인 계획은 무엇인가
- 어디가 밀리고 있는가
- 내가 관여해야 할 것은 무엇인가

이 세 가지를 3초 안에 인지하지 못하면  
일정표의 존재 이유는 약해진다.

## 2. 정보 밀도는 UX의 적이 아니라 설계 대상이다

일정표 UX에서 가장 흔한 오해는  
정보를 줄이면 UX가 좋아진다는 착각이다.

문제는 정보의 양이 아니라 정보의 배열 방식이다.

그래서 다음 기준을 세웠다.

- 기본 View에서는 결정에 필요한 정보만 노출
- 나머지는 인터랙션을 통해 점진적으로 드러남
- 모든 메타데이터는 보여주기 위함이 아니라  
  의사결정을 돕기 위해 존재해야 한다

그 결과 역할이 분리되기 시작했다.

- 프로젝트, 모듈, 기능 트리는 구조 파악용
- 기간 바(Gantt)는 흐름 파악용
- 상태(stage)는 위험 신호 탐지용

## 3. 한눈에 보인다는 말의 실체

그동안 막연하게 사용하던 표현을 다시 정의했다.

한눈에 본다 = 스크롤 없이 모든 정보를 본다 (아니다)  
한눈에 본다 = 판단에 필요한 신호를 즉시 감지한다 (맞다)

그래서 UI 기준을 이렇게 바꿨다.

- 색상은 상태 변화에만 사용
- 텍스트는 의미 전달에만 사용
- 인터랙션은 행동 유도에만 사용

보는 것에서 이해로,  
이해에서 행동으로 자연스럽게 이어지는 흐름을 만든다.

## 4. 사용성은 기능 수가 아니라 행동 비용이다

일정표가 잘 쓰이지 않는 이유는 명확하다.

- 수정하기 번거롭다
- 상태 변경에 맥락 전환이 크다
- 작은 변경에도 피로가 크다

그래서 사용성을 이렇게 정의했다.

사용성이 좋다 = 하고 싶은 행동을 생각보다 적은 에너지로 끝낼 수 있다

이를 기준으로 다음 방향을 잡았다.

- Hover로 임시 계획 생성
- Drag로 기간 조정
- 클릭으로 최소한의 상세 편집

행동 단계를 줄이는 것이 곧 UX 개선이라는 판단이다.  
그리고 결국 내가 직접 써보면서 개선해야, 실질적인 사용성이 나온다.
