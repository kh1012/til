# 위클리 스크럼을 위한 '개인 대시보드' 기능 구현

팀원 개개인에게 어떤 효용가치를 줄 수 있을까? 그리고 실질적으로 이게 쓰임이 있게 하려면 무엇을 더 넣어야할까? 라는 고민에서 시작!

**MyDashboardView를 코드 단위로 나누는 것 자체가 목적이 아니라**,  
“내 작업 데이터가 어떤 흐름으로 요약·해석되어 화면에 표현되는가?”를 명확히 하기 위한 구조 재정비였다.

즉, UI·로직을 분리한 이유는 단순한 리팩토링이 아니라,  
**개인 대시보드가 하나의 분석 시스템으로 동작하기 위한 토대**를 만들기 위한 선택이었다.

## 1) 개인 대시보드를 ‘측정·해석 시스템’으로 바라보기

초기 MyDashboardView는 하나의 React 파일 안에서
- 데이터 수집
- 상태 정리
- 통계 계산
- UI 표현  
  이 전부 뒤섞여 있었다.

이 구조의 문제는 성능이 아니라 **해석 가능성(interpretability)** 이었다.

내가 어떤 활동을 했고, 그 활동이 어떤 패턴을 만들고 있는지,  
그리고 그 패턴이 팀 운영과 내 학습 루틴에 어떤 의미를 갖는지,  
이걸 보기 좋게 만들어야 하는데 그 목적을 가로막는 구조였다.

그래서 오늘의 작업은 “리팩토링”이 아니라 **대시보드의 역할을 다시 정의한 과정**에 가까웠다.

## 2) 역할별 분리: 설계 의도는 유지하되, 해석 여지를 넓히기 위한 구조화

### ⬦ Before
모든 책임이 MyDashboardView에 몰려 있어  
“무엇이 데이터 해석이고 무엇이 UI인지” 경계가 사라진 상태였다.

### ⬦ After (의도 중심 설명)

```
my/
├── MyDashboardView.tsx        # 화면 조합만 담당 (표현 레벨)
├── hooks/
│   └── useMyDashboard.ts      # 데이터 해석(필터링·집계·상태관리 중심)
├── utils/
│   └── dashboardUtils.ts      # 순수 계산식 (통계, 기간 연산)
└── components/                # 시각화 요소
├── StatCard.tsx
├── DomainPieChartCard.tsx
├── DashboardFilters.tsx
├── TrendChart.tsx
├── ProjectBarChart.tsx
├── ProgressOverview.tsx
└── RiskSummary.tsx
```

이 구조로 바꾼 이유는 단순히 관심사를 나누는 게 아니라  
**개인 활동 데이터를 해석하는 흐름을 분리하여, 각 단계에서 더 깊이 있는 개선이 가능하게 하기 위함**이다.

## 3) useMyDashboard: “데이터 해석 엔진”을 따로 두는 전략

오늘 가장 큰 변화는 **Custom Hook이 단순 상태 저장소가 아니라 분석 엔진 역할을 수행**하도록 만든 점이다.

여기에 담긴 핵심 기능은 다음과 같다:

- 특정 회원 선택에 따른 활동 필터링
- 기간 기반 트렌드 계산
- 도메인/프로젝트 기반 분류
- 통계 계산 (예: 아이템 수, 완료 패턴, 도메인 분포 등)

이 작업은 사실상 UI 렌더링과 별개다.  
**대시보드의 본질은 시각화가 아니라, “해석 가능한 데이터 구조를 만들어 전달하는 것”이기 때문.**

덕분에 지금은 UI를 교체하더라도 해석 로직이 흔들리지 않는다.

## 4) utils: 순수 함수 분리는 단순 ‘깔끔함’을 위한 게 아니다

`dashboardUtils.ts`에 계산식을 분리한 이유도 명확하다.

- UI나 React 훅의 영향 없이 검증 가능해야 한다.
- 기간 연산, 통계 계산, 태깅 규칙 등은 장기적으로 재사용 가능하다.
- 무엇보다 **로직을 문서화할 필요가 있을 때**, UI와 얽혀 있으면 분석 자체가 어려워진다.

즉, 이 선택은 **대시보드의 확장성을 위한 투자**다.  
(특히 나중에 "팀 전체 대시보드"와 결합할 때, 이 분리는 큰 이점이 된다.)

## 5) Barrel Export로 컴포넌트 계층을 깨끗하게 유지

UI 컴포넌트 파일이 많아질수록 import 경로가 지저분해지는데,  
이걸 `components/index.ts`로 정리하니 관리 비용이 대폭 줄었다.

이건 작은 변화지만, **컴포넌트를 데이터보다 먼저 읽히게 하지 않는다는 구조적 목적**이 있다.  
(즉, UI가 중심이 아니라 ‘데이터 흐름이 중심’이 되도록.)

## 6) 결과: 더 이상 ‘대시보드 화면’을 개발하는 것이 아니라
### **내 작업 패턴을 분석하는 알고리즘을 개발하는 느낌**

코드 분리는 단순한 정리 작업이 아니었다.  
대시보드의 본질을 다시 잡고, 앞으로 확장할 수 있는 해석 기반 구조를 만든 변화였다.

- 파일 책임이 명확해짐
- 분석 로직이 눈에 보이고 검증 가능해짐
- UI 교체가 더 쉬워짐
- 대시보드가 “내 활동에 대한 관찰 도구”로 탈바꿈

MyDashboardView.tsx를 열었을 때 **“여긴 그냥 화면만 조합하는 곳”**이라는 명확한 느낌이 든다는 건, 방향을 제대로 잡았다는 신호다.

## 내가 가져갈 키워드

- 개인 대시보드 = “데이터 해석 시스템”
- UI보다 해석 로직 분리가 우선
- Custom Hook은 로직 저장소가 아니라 분석 엔진
- 순수 함수는 재사용성과 검증 가능성을 높이는 핵심 축
- Barrel Export로 시각화 계층을 얇고 단정하게 유지
- 대시보드를 구조화하면 “내 작업 흐름을 측정하는 루프”가 명확해진다
