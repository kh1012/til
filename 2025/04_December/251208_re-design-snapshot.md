# Snapshot V2 – 프로젝트 중심 재설계 관점 TIL

스냅샷 V2를 **팀원이 실제로 보고 싶어하는 관점**에서 다시 설계했다  
과거 스냅샷은 “개인 중심 기록”에 가까웠고, 실제 협업 흐름이나 프로젝트 흐름이 잘 드러나지 않았다.  
그래서 오늘은 **프로젝트 중심으로 구조를 재편하고, 기능을 함께 만드는 사람들의 관계·의존성·협업 구조를 데이터로 표현하는 방향**까지 고민했다.

결과적으로 “스냅샷 → 대시보드 → 팀 운영 체계”로 이어지는 흐름을 강화했고,  
특히 **팀원이 스스로 무엇을 보고 싶어하는가**를 기준으로 정의를 바꿨다는 점이 결정적 변화다.

## 1) 기존 스냅샷의 한계가 분명했다

기존 구조는 다음 문제를 가지고 있었다.

- 개인이 어떤 일을 하는지는 보였지만  
  **그 일이 어떤 프로젝트 흐름 속에서 어떤 역할을 하는지 보이지 않았다.**
- 협업 관계가 전혀 드러나지 않아  
  **누가 누구와 함께 어떤 기능을 완성하는지 파악이 불가능했다.**
- Plan / Progress / Next가 있었지만  
  **하나의 기능이 어느 스프린트, 어느 프로젝트 흐름에 위치하는지 연결성이 부족했다.**
- 결국 “업무 보고용 텍스트”로만 소비되고,  
  **팀이 문제를 조기에 감지하거나 리소스를 재배치하는 데 활용되기 어려웠다.**

즉, 개인 기록은 존재했고, 데이터도 있었지만,  
“팀이 쓰기 위한 구조”는 아니었다.

## 2) 그래서 관점을 ‘개인 중심 → 프로젝트 중심’으로 강제 전환했다

오늘 대화에서 본질적으로 다룬 포인트는 이것이다.

> “개인의 일이 아니라 **프로젝트의 진행 상황**을 중심으로 데이터를 다시 구조화해야 한다.”

이 한 줄이 스냅샷 V2를 완전히 바꿨다.

새로운 관점은 다음과 같다.

### 프로젝트가 1차 기준이 된다
- 프로젝트 → 모듈 → 기능 단위로 자연스럽게 보이게 하고
- 그 안에서 누가 어떤 역할을 맡고 있는지가 따라오게 한다.

### 개인은 ‘프로젝트 안에서의 위치’로 표현된다
- 개인을 중심으로 묶던 구조 대신  
  **프로젝트를 중심으로 사람들을 나열하는 방식**으로 재편.

### 협업자(Collaborator), 의존성(Dependency)을 1급 데이터로 올렸다
- “누구와 함께 만드는 기능인가?”는 실제 협업에서 가장 중요한 정보인데  
  기존 스냅샷에서는 무시되고 있었다.
- 그래서 이번 수정에서 **협업 구조를 스냅샷 핵심 요소로 승격**했다.

이 관점 전환 자체가 이미 팀 운영 체계의 질을 한 단계 끌어올린다.

## 3) 팀원이 ‘보고 싶은 것’을 기준으로 데이터 구조를 다시 정의했다

오늘 논의에서 가장 중요한 변화는 “팀 기준”으로 다시 생각했다는 점이다.

팀원 입장에서 보고 싶은 정보는 다음과 같다.

1) **내가 속한 프로젝트의 전체 흐름**
2) **나와 함께 기능을 만드는 사람들**
3) **이번 주에 프로젝트 진행률을 끌어올린 핵심 작업**
4) **다음 주에 무엇이 이어질지**
5) **어디에서 리스크가 발생하고 있는지**
6) **리소스가 쏠린 곳은 어디인지**

즉, 팀원도 스냅샷을 통해  
“우리가 지금 무엇을 만들고 있는가”  
“무엇이 막혀 있고 어디가 밀리고 있는가”  
“이번 주 팀의 흐름은 무엇이었는가”  
를 알고 싶어 한다.

그래서 기존 텍스트 기반 보고 구조를 완전히 버리고,  
이를 **UI에서 바로 읽힐 수 있는 데이터 스키마**로 설계했다.

## 4) Snapshot V2 핵심 구조(스키마) – 오늘 대화에서 정리된 형태

### 단일 작업이 아닌 “프로젝트 기반 구조화”
- domain
- project
- module
- feature
- next[] { text }
- risk[]
- riskLevel
- collaborators
 
이걸 UI에서 제대로 표현하려면 결국 JSON 스키마가 바뀌어야 하고,  
Cursor V2에게 이 스키마를 기준으로 파일 변환을 요청하는 파이프라인도 개편해야 한다.

오늘 그 전체를 다 설계했다.

## 5) 개인 대시보드와의 연결 — 팀원의 시야를 넓히는 방향으로 재정의

기존 개인 대시보드는 "내가 무엇을 했는지 보는 용도"였다.  
하지만 오늘 논의로 관점이 크게 달라졌다.

### 새로운 목적
- **팀 전체 흐름 속에서 개인의 위치를 확인하는 도구**
- **내가 맡은 기능이 프로젝트 전체의 어느 지점인지 시각적으로 보여주는 도구**
- **협업 관계와 의존성을 시각적으로 연결해주는 도구**
- **리소스 쏠림 여부를 감지하는 인사이트 제공**

즉, 더 이상 개인 대시보드는 개인용 생산성 도구가 아니다.  
이제는 **팀 운영 체계를 구성하는 핵심 컴포넌트**가 된다.

## 6) 오늘의 핵심 인사이트

1. **보고 체계는 “어떻게 보고하는가”가 아니라 “무엇을 보고 싶은가”를 기준으로 만들어야 한다.**
2. 개인 중심 구조는 협업을 드러내지 못한다. 프로젝트 중심으로 재편해야 한다.
3. 협업자·의존성·프로젝트 흐름은 팀 운영에서 필수 데이터이며 1급 요소로 승격해야 한다.
4. 스냅샷은 더 이상 “보고서”가 아니라 “팀의 실시간 운영 데이터”가 된다.
5. 개인 대시보드는 개인 도구가 아니라 팀 운영의 일부다.
6. 이 구조화가 이루어져야 “팀이 스스로 문제를 감지하는 시스템”이 작동한다.
