# tmux로 클로드 팀 구성하기 — 오케스트레이터/워커 패턴의 장단점

> 한 사람이 여러 클로드를 동시에 다루기 위해 tmux를 그대로 "팀의 사무실 배치도"로 쓰는 방식의 실전 트레이드오프.

## 배경

frontend-3 프로젝트에서 두 갈래 작업을 동시에 굴려야 했다. Design Notebook 디테일 뷰 4건과 응답 스트리밍 / HITL 개선 2건. 하나의 클로드 세션 안에서 이 6개를 직렬로 처리하면 컨텍스트가 금세 오염되고, 복수 세션을 따로 띄우자니 사용자(나) 입장에서 "어디에 뭘 시켰는지" 추적이 안 된다.

그래서 tmux 세션 `do`를 다음 형태로 짰다.

```
do
├── orchestrator   (1 pane,  %0)        ← 사용자 ↔ 클로드 본인
├── detail-view    (4 panes, %1 %3 %2 %4, 2x2 tiled)
│     #1   #2
│     #3   #4
└── streaming-hitl (2 panes, %5 %6, 좌/우)
      #1   #2
```

규칙은 단 하나. 사용자는 **`orchestrator` pane(`%0`)에만** 말하고, 거기 있는 클로드(나)가 워커 pane들에게 작업을 분배한다. 워커는 각자 독립된 클로드 세션에서 `/pdca plan → do → analyze → report → /commit-kr` 사이클을 돈다.

설치는 한 줄로 끝난다.

```bash
tmux new-session -d -s do -n orchestrator -c "$PROJ"
tmux new-window  -t do: -n detail-view -c "$PROJ"
tmux split-window -t do:detail-view -h -c "$PROJ"
tmux split-window -t do:detail-view.0 -v -c "$PROJ"
tmux split-window -t do:detail-view.2 -v -c "$PROJ"
tmux select-layout -t do:detail-view tiled
# … streaming-hitl 윈도우도 같은 식으로
tmux set-option  -t do pane-border-status top
tmux set-option  -t do pane-border-format ' #{pane_index}: #{pane_title} '
tmux select-pane -t do:detail-view.0 -T "detail-1"  # 라벨
```

이 패턴을 며칠 굴려보면서 정리한 장단점이 이 글의 본론이다.

## 핵심 내용

### 라우팅 키는 "pane index"가 아니라 "pane id(%N)"

tmux 분할은 `index`가 분할 순서대로 매겨져서, 시각적 위치(좌상/우상/좌하/우하)와 인덱스가 어긋나기 쉽다. 실제로 위 구성에서 `detail-view`의 인덱스는 `0, 1, 2, 3`인데 ID는 `%1, %3, %2, %4` 였다.

오케스트레이터 입장에서 워커를 호출할 때 인덱스(`do:detail-view.1`)를 쓰면 이후 split/swap 한 번으로 라우팅이 깨진다. 반면 `%3` 같은 **pane ID는 세션 종료 전까지 변하지 않는다**. 그래서 분배 매핑은 ID로 잡는다.

```
detail-1 → %1
detail-2 → %3
detail-3 → %2
detail-4 → %4
stream-1 → %5
stream-2 → %6
```

지시 송신은 다음과 같이 한다.

```bash
tmux send-keys -t %3 '/pdca plan design-notebook-detail-2 ...' Enter
```

### 장점

**1. 컨텍스트 격리.** 워커 #1의 토큰 누적, 잘못된 파일 탐색, 막다른 길 시도가 워커 #2에게 전염되지 않는다. 한 워커의 컨텍스트가 망가지면 그 pane만 죽이고 다시 띄우면 된다. 단일 세션에서 6개 작업을 직렬로 돌릴 때 한 번 잘못 든 길이 이후 모든 작업의 정확도를 갉아먹던 문제가 사라진다.

**2. 진짜 병렬성.** 4개의 디테일 뷰 개선이 실제로 **동시에** 진행된다. 디스크/CPU가 받쳐준다는 전제 하에, 직렬 대비 6배 빠른 시간 압축이 가능하다.

**3. 책임이 pane과 1:1.** "디테일 #2 어디까지 갔지?"의 답이 항상 `%3` 한 군데에 있다. 화면 분할 자체가 칸반 보드처럼 동작한다.

**4. 메타 시야 확보.** 오케스트레이터에서 `tmux capture-pane -t %3 -p | tail -20`로 워커 출력만 떼어와 진행 상황을 본다. 직접 attach해서 들여다보지 않아도 된다.

**5. 재현 가능한 셋업.** tmux 명령들은 그대로 셸 스크립트가 된다. 작업이 바뀌어도 윈도우 이름과 pane 수, 라벨만 바꾸면 다음 날 같은 형태를 한 줄로 띄운다.

**6. attach/detach 자유.** 자리를 비워도 워커는 계속 돈다. 노트북을 닫고 회의에 들어갔다가 와도 클로드들은 자기 사이클을 끝내놓는다. 이건 같은 일을 단일 IDE/터미널 창으로 했다면 불가능했다.

**7. 작업 단위 가시성.** pane border에 `[detail-2: do]` 같은 식으로 단계를 라벨링해두면 6개 워커의 PDCA 단계가 한눈에 보인다.

### 단점

**1. 같은 파일 동시 수정 충돌.** 가장 비싼 함정이다. 디테일 뷰 4개가 모두 같은 디자인 토큰 파일이나 같은 컴포넌트 베이스를 만지면, 가장 늦게 커밋한 워커가 앞 워커의 변경을 덮어쓰거나, 머지 충돌로 멈춘다. 분배 시점에 **파일 경계로 작업을 쪼개는 것**이 사실상 필수 조건이다. 영역이 겹치는 작업이라면 이 패턴 자체가 부적합하다.

**2. 폴링 토큰 비용.** 오케스트레이터가 진행 상황을 알기 위해 `capture-pane`을 주기적으로 읽으면 그만큼 토큰을 태운다. 6개 pane을 1분마다 본다면 한 시간에 360번. 이걸 줄이려면 워커가 끝났을 때 자기 입으로 `tmux send-keys -t %0 '워커-N 완료' Enter` 신호를 보내게 만들어야 한다(이걸 워커 프롬프트에 박아두는 게 핵심).

**3. 결정 일관성 깨짐.** 워커들끼리 서로의 결정을 모른다. 워커 #1이 "스트리밍 chunk를 타입 X로 정의"하고 워커 #2가 "타입 Y로 정의"하면 두 PR이 합쳐질 때 충돌한다. 오케스트레이터가 **공통 결정은 미리 못 박아 분배 프롬프트에 넣는 것**이 유일한 방어책이다.

**4. 로컬 자원 충돌.** `yarn install`이 두 pane에서 동시에 돌면 lock 파일이 깨진다. `yarn build`도 마찬가지. 그래서 전역 메모리에 **"/max-do에서 빌드/테스트 금지, /max-qa에서만 실행"** 룰을 박아둔 게 결정적으로 도움이 된다. dev server는 한 pane만 띄우고 나머지는 그것을 공유한다.

**5. 모니터링 인지 부하.** pane 7개를 한 사람이 동시에 본다는 건 사실상 불가능에 가깝다. 4 + 2 = 6 워커가 상한선이라고 느껴졌다. 그 이상이면 오케스트레이터 클로드도 "어느 워커가 무엇을 하는지" 헷갈리기 시작한다.

**6. HITL 단절.** 가장 큰 트레이드오프. 워커가 모호한 결정을 마주쳤을 때 사용자에게 직접 물어볼 수 없다. 오케스트레이터를 거쳐야 한다. 메모리에 `commit까지만, push/PR은 사용자가 수동` 룰이 박혀있는 이유가 바로 이것이다. 워커가 자율적으로 push/PR까지 가버리면 사용자가 끼어들 타이밍 자체가 사라진다.

**7. 디버깅 도미노.** 한 워커가 멈추면(예: 권한 프롬프트 대기, 네트워크 hang) 그 워커에 의존하는 후속 작업이 줄줄이 막힌다. tmux는 이걸 시각적으로 보여줄 뿐, 자동으로 풀어주지 않는다. 오케스트레이터가 주기적으로 "멈춘 워커"를 찾아서 깨우거나 죽여야 한다.

**8. 인스턴스 자원.** 클로드 세션 6~7개가 한 머신에서 돈다. 메모리 관점에서는 큰 부담은 아니지만, 토큰 관점에서는 단일 세션 대비 합산 사용량이 크다. 비용/quota를 의식해야 한다.

### 언제 쓰고 언제 안 쓰나

쓸 만한 조건:
- 작업 6건 이하, 파일 경계로 명확히 쪼갤 수 있음
- 공통 결정(타입, 토큰, 인터페이스)이 이미 결정되어 있음
- 사용자가 push/PR 단계에서만 개입하면 충분한 작업

피해야 할 조건:
- 같은 컴포넌트/같은 함수를 여러 워커가 만져야 함
- 작업 중간에 사용자 결정이 자주 끼어들어야 함(디자인 협의, 스펙 변경)
- 빌드/테스트가 작업 사이클에 묶여 있고 lock 충돌이 불가피함

## 정리

tmux는 "여러 클로드를 사람 한 명이 다루는 UX"를 거의 공짜로 제공한다. pane이 곧 워커, ID가 곧 라우팅 키, border가 곧 칸반 라벨. 별도 툴 없이 이렇게까지 갈 수 있다는 게 매번 신기하다.

대신 이 패턴은 **작업이 파일 경계로 깨끗하게 쪼개진다**는 가정 위에서만 효율이 나온다. 그 가정이 깨지면 단일 세션의 직렬 처리가 오히려 빠르고 안전하다. "팀을 만들 가치가 있는 일인가"를 분배 전에 한 번 더 묻는 게, 이 패턴을 잘 쓰는 유일한 길 같다.

오늘 메모해 두고 싶은 한 줄: **인덱스 말고 ID로 라우팅하기, 폴링 말고 완료 신호 받기, 그리고 "겹치는 작업"은 분배하지 않기.**
