# 6-pane tmux 오케스트레이터로 디자인 노트북 디테일뷰 9개를 동시 정합한 하루

> 단일 클로드 세션으로는 따라갈 수 없는 양의 디자인 폴리싱·회귀 수정·구조 분석을 6개 워커 pane으로 분산하면서 익힌 워크플로우. 분배 규칙·자동 commit·관점 전환·충돌 회피를 한 사이클 안에 흘리듯 운영한 기록.

## 배경

Step 1~9 디테일뷰의 원본(prototype)과 현재(frontend-3) 사이 갭을 PDCA Iterate 루프로 좁히는 작업이 잡혔다. 처음엔 한 번에 4개 Step을 분석하고 90%까지 끌어올리는 게 목표였는데, 디자인 폴리싱·인터랙션 회귀·shared 재사용 정책 같은 "끝없는 미세 작업"이 누적되기 시작했다. 단일 세션으론 컨텍스트가 폭발해서 세션을 분할해야 했다.

tmux `do` 세션의 detail-view 윈도우 4 pane + streaming-hitl 윈도우 2 pane = 총 6 워커 + 외부 오케스트레이터 1 = 7 클로드 동시 운영 패턴으로 옮겨갔다. 아래는 그 운영 규칙과 거기서 나온 통찰들이다.

## 핵심 내용

### 1. 분배 정책의 진화 (v1 → v3)

**v1 — 단순 영역 분할**: 처음엔 4 워커에 Step별로 분배했다. Step 1 → %1, Step 3 → %3 같은 식.

**v2 — 정책 위임**: "이후 보내는 프롬프트는 알아서 유휴 pane 분배 또는 큐잉" 명령이 들어오면서 사용자 입력 = 의도, 분배 결정 = 오케스트레이터로 책임 이동.

**v3 — 영역 기반**:

| 상황 | 처리 |
|------|------|
| 같은 파일/영역 작업 진행중 | 같은 pane에 send-keys로 메시지 추가 (컨텍스트 신선) |
| 다른 영역 + idle pane 있음 | 즉시 다른 pane에 분배 (병렬) |
| 다른 영역 + idle 없음 | 큐잉 (트리거 명시) |

핵심은 "파일 충돌 가능성"을 매번 판단하는 것. 같은 파일을 두 워커가 만지면 git diff 충돌이 거의 확정이라, 사용자가 명시적으로 "별도 pane으로 빼"라고 해도 영역이 겹치면 큐잉이 안전하다.

### 2. PDCA Gap 분석에서 관점이 바뀌는 순간

처음엔 "원본 코드와 100% 동일하게" 라는 기준으로 분석했다. 결과: Step 1 36%, Step 3 25%, Step 4 60%, Step 5 32%. 충격적으로 낮았는데, 사용자가 한마디 던지면서 관점이 뒤집혔다:

> "구조 자체를 완전히 똑같이 할 필요는 없는데, 현 frontend-3의 코드 베이스에 맞춰 프론트엔드 쪽 UI만 구성하고, 실제 데이터는 백엔드가 하나씩 채워줄거야. 인덱스DB가 다르다 이런 건 코드 상에 대응되는게 있다면, gap이 없다고 보는게 맞을거 같아. 이 관점으로 다시 본다면?"

**관점 전환의 본질** = "백엔드가 채워줄 데이터를 받을 UI 슬롯이 frontend-3에 있는가" 만 본다. 인프라(IndexedDB persistence, Zustand store, 실 API 클라이언트, 공학 계산)는 Gap에서 제외.

같은 코드를 다시 분석하니 Step 1 36% → 77%, Step 4 56% → 56% (비슷), Step 5 32% → 38%. **Step 3는 오히려 6%로 더 낮아졌는데** 이유는 "사용자 편집 UI"가 시나리오 핵심인데 frontend-3에 거의 전무하기 때문. 즉 관점이 바뀌면 새 갭이 보인다.

비공식 교훈: **Gap 분석의 분모를 명시하지 않은 매칭률은 의미가 없다.** "구조 매칭률 36%"는 "UI 슬롯 매칭률 77%"와 다른 척도. 분석 의뢰자에게 분모를 정의해서 합의받지 않으면 분석의 가치가 떨어진다.

### 3. 사이클 완료 신호 → 자동 분할 commit

처음엔 변경사항이 30+ 파일 쌓이도록 commit이 한 번도 안 일어났다. 사용자가 "중간중간 commit 정리해야 하는데 변경사항만 쌓이는 거 같네?" 하고 지적하면서 정책 추가:

```
사이클 완료 신호 도착 → 영역별 conventional commit 자동 실행
```

이때 핵심은 **"진행중 워커 영역 파일은 commit에서 제외"** 하는 것. 같은 파일을 다른 워커가 만지는 중에 git add 하면 일관성 깨짐. 그래서 add는 명시적 파일 경로로만, glob도 진행중 영역과 격리해서 사용.

```bash
# 좋은 예
git add src/features/design-notebook/ui/detail/forces/ \
        src/features/design-notebook/lib/notebook-forces-utils.ts

# 나쁜 예 — 다른 워커 변경분도 휩쓸려 들어감
git add -A
```

세션 끝까지 누적 commit 60+. 분할이 잘 됐는지는 git log를 보면 영역별로 한 줄 conventional message로 깔끔히 떨어진다. 회고할 때 "어디서 무엇을 했는지" 추적이 쉽다.

### 4. 회귀가 패러다임 전환의 신호일 때

Step 2 부재 판단(3D)의 selection 시각 위계는 **외곽선 강조 → fill accent**로 한 번에 안 바뀌었다. 4번 회귀:

| 회귀 | 증상 | 원인 |
|------|------|------|
| 1차 | hover 시 색이 너무 연함 | 외곽선 색상 토큰 약함 |
| 2차 | 슬라브 hover만 부글거림 | z-fighting (반투명 default + outline overlap) |
| 3차 | 보/기둥/벽체 선택 안 됨 | raycast 차단 (outline mesh가 raycast 가로챔) |
| 4차 | 슬라브는 select fill 잘 되는데 보/기둥/벽체는 안 보임 | meshStandardMaterial × lighting 영향 (별도 TIL 참조) |

각 회귀를 잡을 때마다 사용자가 패러다임을 살짝씩 흔들었다. "외곽선 두께 두껍게" → "테두리 말고 fill로" → "기본 투명도 빼고 hover만 투명 fill로" → "슬라브 hover 색상이 보·기둥과 다른데?" 식으로.

비공식 교훈: **"회귀 4번이 누적되면 그건 회귀가 아니라 패러다임 전환이다."** 첫 회귀에서 "외곽선 강조"를 고집하지 말고 사용자의 실제 의도(fill로 강조하고 싶음)를 빨리 캐치해서 패러다임 자체를 바꾸면 작업량이 줄어든다.

### 5. shared/ 재사용 정책의 자연 진화

세션 중반쯤 사용자가 "각 컴포넌트, 개별로 만들지 말고 shared/ui/primitive에 재사용할 수 있으면 적극 이용해줘" 라고 못박자 분위기가 바뀌었다. 그전엔 워커들이 design-notebook 안에 새 컴포넌트를 막 만드는 경향이었는데, 이후엔 신규 컴포넌트가 shared/ 레이어에 신설되기 시작했다:

- `shared/ui/layout/FloatingPanel` — figma 스타일 floating + collapse 헤더 토글
- `shared/ui/form/SearchInput` — Cmd+K 패턴 (검색 아이콘 + autoComplete off + tight leading)
- `shared/ui/feedback/LoadingOverlay` — backdrop-blur + LoadingSpinner + radius inherit

이것들이 한 번 만들어지면 다른 디테일뷰들이 자연스럽게 import해서 쓰게 된다. **shared/ 레이어는 "발견되는" 게 아니라 "필요할 때 추출되는" 것**이라는 걸 다시 확인했다. 처음부터 모든 shared 컴포넌트를 만들 순 없고, 두 번째 사용처가 나타날 때 추출하는 게 자연스럽다.

### 6. 작업 큐잉 vs 즉시 분배의 trade-off

사용자가 한 번에 여러 요청을 던질 때 (특히 5분에 7~8개) 즉시 분배 vs 큐잉 결정이 매번 갈렸다. 즉시 분배의 위험: 같은 파일을 두 워커가 만지면 commit 시 충돌. 큐잉의 위험: 사용자 체감 속도 느려짐.

규칙으로 정리하니:
- **충돌 위험이 명백하면 무조건 큐잉** (트리거 조건 명시)
- **충돌 모호하면 같은 pane 통합** (send-keys로 추가 메시지)
- **명백히 다른 영역이면 즉시 다른 pane**

이게 세션 후반에 자리잡으면서 사용자가 "큐잉 리스트 보여줘" 라고 물어봤을 때도 즉시 답변 가능했다. **큐 가시성**이 분배 시스템의 신뢰성을 좌우한다.

### 7. 워커가 완료 신호를 안 보낼 때

세션 끝자락에 사용자가 "끝났나?" 라고 물었는데 %3가 작업은 끝났지만 완료 신호 trigger 메시지를 안 보냈다 (워커가 깜빡함). capture-pane으로 입력창이 비어있고 작업 표시가 사라진 걸 확인 → 신호 미수신이라도 idle 상태면 commit 진행.

**완료 신호 의존 ≠ 절대적**. 워커가 신호를 빼먹을 수 있으니 capture-pane으로 실제 상태를 주기적으로 검증하는 게 안전. 자동화의 함정 — 신호에만 의존하면 신호가 빠질 때 시스템이 멈춘다.

## 정리

- 6-pane 오케스트레이션은 **분배 결정의 자동화 + 자동 분할 commit + 사용자 가시성** 3가지가 동시에 갖춰질 때 비로소 효율이 나온다. 한 가지만 빠져도 사용자가 "지금 뭐 하고 있어?" 라고 물어봐야 하는 상황이 반복된다.
- Gap 분석은 분모(척도)부터 합의해야 한다. "구조" vs "UI 슬롯" vs "사용자 시나리오" 같은 분모 차이가 매칭률을 30%p 이상 흔든다. 의뢰자가 기대하는 분모를 추측 말고 명시적으로 묻자.
- 회귀가 3번 이상 같은 영역에서 반복되면 그건 디버그가 아니라 디자인 패러다임 변경 신호다. 빨리 인정하고 패러다임 자체를 갈아엎자.
- shared 컴포넌트는 두 번째 사용처가 보일 때 추출하는 게 자연스럽다. 처음부터 만들면 over-engineering, 너무 늦게 추출하면 중복 누적.
- 워커가 신호를 빼먹는 건 비정상이 아니라 정상적 분포 안에 있는 일이다. 신호를 trust하지 말고 verify하는 보조 채널(capture-pane, git status)을 항상 붙여두자.
- tmux `send-keys` 후 Enter가 한 번에 안 들어가는 케이스가 종종 있다. 큰 프롬프트는 `sleep 2` 후 별도 Enter로 분리해서 보내는 게 안정적.
- 한 세션에 누적 60+ commit이 가능하다는 걸 처음 확인했다. 분할 commit은 더 자주 하는 게 좋다 — 한 번 묶이면 분리하기 어렵다.
