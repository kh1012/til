# Frontend Architecture

아키텍처에 대해 깊이 고민하고 시작한 프로젝트가 거의 없었다.  
당장 만들어서 당장 서빙하는게 주요한 과제였다.  
여유가 생긴김에 프론트엔드 아키텍처에 대해 고민해본 기록을 남겨보려 한다.  

![FE_Architecture](https://raw.githubusercontent.com/kh1012/til/main/assets/260222_FE_architecture.png)

## 모니터링을 제외한 주요 요소

우선 크게 7가지로 분류해봤다.  
물론 좋은 아키텍쳐 방법론들이 많겠지만, 실무에서 직접적으로 내가 그릴 수 있는 수준은 위와 같을거라 생각한다.  

> 1) Design System
> 2) Quality Assurance
> 3) Performance
> 4) Directory Structure
> 5) Framework
> 6) Validation
> 7) State Management

아마 추가로, monitoring이 있겠지만, 이건 제외했다.  
프로젝트 초반에 필요한 건 빌드 타임의 안정성과 런타임의 안정성, 그리고 유지보수성이다.  
Sentry를 좀 다뤄봐야하는데, 이건 별도 기록으로 남겨둬야 겠다.  

### 1) Design System

디자인 시스템은 시멘틱 토큰과 유동적인 시스템으로 분류 하였다.  

#### 시멘틱 토큰

tailwindcss를 쓰다보면, 자연스럽게 기본적으로 정의된 형태를 쓰기 마련이다.  
하지만 기획자나 디자이너가 원하는 요구사항을 최소한의 변경으로 구현하기 위해서는 맥락 단위의 시멘틱 토큰이 필수이다.  

#### 유동적인 시스템

디자인 시스템이 고정되어 있다면, 반응형을 구현하기는 쉽지 않을거라 생각한다.  
물론 device의 width에 따라 제각각 구분되는 형태를 구현해도 되지만, 매끄러운 사용자 경험을 위해서는 유동적인 시스템이 필요하다.  
Google 디자인 가이드와 MDN 문서를 읽어보면 최신 트랜드는 320px - 1440px 정도의 보간 값이 필요하다.  
우아하게 CSS의 clamp를 이용하면 이 구간내에 선형보간 되는 여백과 간격, 레이아웃, 폰트의 자간 등을 선형적으로 구현할 수 있게 된다.  

### 2) Quality Assurance

QA는 테스트 트로피와 ESLint의 아주 기본적인 RULE로 시작 해본다.  
ESLint는 하나씩 쌓아가면서 프로젝트를 참여한 인원들의 합의를 기반으로 하나씩 쌓아가는게 맞아보인다.  
최소한 디렉토리 구조의 응집도와 단방향 의존성을 위한 조치, 디자인 토큰에 위배되지 않는 임의의 값은 보장해야할거 같아 우선 2가지로 구성 해뒀다.  

#### 테스트 트로피

테스트 구조는 총 4단계로 구성된다.  
이름 모를 어떤 아저씨가 했다는데, Kent C. Dodds 였나? 
> 테스트 트로피(Testing Trophy)는 Kent C. Dodds가 제안한 테스트 전략 모델로, 테스트의 비용 대비 효율성에 집중한다. 기존의 테스트 피라미드가 단위 테스트(Unit Test)의 양적 확보에 집중했다면, 테스트 트로피는 **통합 테스트(Integration Test)** 를 가장 핵심적인 부분으로 본다.

나 역시도 기존까지는 유닛 테스트가 많으면 컴파일/런타임에 안정성이 확보될거라 생각했었는데,  
여러 단위를 조합하는 통합 테스트에 대해서는 깊게 생각해보지 않았던 것 같다.  
이 고민을 하는것도 또 다른 즐거움일지도...

#### ESLint

lint는 서두에 이야기 했듯이, 간단하게 2가지만 정의한다.
- no-restricted-imports
- no-arbitrary-values

모듈 참조 관계를 강제하고, 임의의 값을 제한하는 것만으로도 충분히 초반에 서로 다른 길을 가는 것을 방지 할 수 있다.  

### 3) Performance

이쪽은 모니터링과 연관이 있어 보이긴 하나,  
당장 모니터링(Sentry)는 다루지 않을거라 우선 넣어 두었다.  
크게 3가지 관점이 있는 것 같다.  

1. 성능 최적화
2. 빌드타임 최적화 및 체크
3. 런타임 최적화 및 검증

#### 성능 최적화

성능 최적화의 경우에는 여러 방안들이 있겠지만, 우선 고려해야 하는 건 2가지로 보인다.  
사실 나머지는 Web Vitals를 측정하면서 하나씩 쌓아가는게 맞아 보인다.  
- 낙관적 업데이트
- 가상화

압도적인 사용성을 가져오려면, 코드 복잡도가 올라가더라도 낙관적 업데이트는 필수처럼 보인다.  
물론 추후에 하드웨어 성능이 올라가거나 네트워크 속도가 점차 빨라진다면, 이 또한 과거 플로피디스켓 시절에 바이트 하나를 줄이기 위해 노력했던 것 처럼, 불필요한 노력이 될 수도 있다.  
하지만 아직까지는 극강의 사용성을 위해서는 피할 수 없는 존재로 보여진다.  

가상화는 참 우아한 것 같다.  
DOM을 최소로 유지하면서, 사용자에게는 무한한 리스트를 보여줄 수 있다.  
사용자는 마치 새로 생성되는 것 처럼 느끼지만, 실질적으로 제한된 노드로 기가막힌 효과를 낼 수 있다.  

#### 빌드타임 최적화 및 체크

Nextjs 16버젼부터는 Turbopack이 기본으로 채택되었다.  
이로 인해 빌드 속도가 비약적으로 빨라졌고, 그렇다면 개발 단계에서의 최소 성능지표는 준수해야 한다.  
@LHCI 라이트하우스 CLI를 이용해서 최소 성능지표를 측정하고, 이를 통과하지 못하면 빌드 자체가 실패하도록 구성할 수 있다.  
CI/CD 수준에서 ESLint와 조합하여 성능 수치를 만족하지 못하면 배포를 실패하도록 구성한다면 얼마나 아름다운가!  
사람의 실수는 줄이고, 개발자가 미처 고려하지 못했던 성능을 자동으로 챙길 수 밖에 없다.  
개발자 스스로에게도 좋은 고민거리가 될거 같다.  

#### 런타임 최적화 및 검증

Sentry Performance를 이용해서 런타임 성능을 측정할 수 있다고 한다.  
다만 아직 이 부분을 경험해보지 못해서 새로운 프로젝트에는 반드시 적용해보고 그 결과를 정리해둬야 겠다.  
(다른 툴 OpenTelemetry도 있다고 하는데, 요건 사이드로 해봐야지!)

### 4) Directory Structure

요건 좀 고민이 많다.  
사실 아키텍쳐론을 들여다보면, 장단점이 많은데, 유독 눈에 띈 형태가 FSD이다.  
사실 컴포넌트를 설계하다보면 하나의 디렉토리에 무수히 많은 파일들을 생성하게 된다.  
디렉토리를 바라보며 디버깅을 해보려고 하면, 어느 디렉토리를 건드려야 할지 당황할때가 종종 있다.  
특히, 내가 작성한 코드가 아니라면 더욱이..  
그래서 도메인 단위로 디렉토리를 구성하는 FSD가 매력적으로 다가왔다.  
하나의 도메인을 선택하면 그 도메인 하위에 내가 알고 싶은 파일들이 모두 담겨있다.  
응집도는 올라가고, no-restricted-imports 규칙과 함께하면 결합도는 낮출 수 있다.  

### 5) Framework

기존까지는 Nextjs 15를 무지성으로 써왔다.  
Nextjs 16부터는 Turbopack이 기본으로 채택되었다.  
그리고 React 19.2를 기본으로 채택하여 React Compiler를 쓸 수 있다.  
아주 간단하게 설정값에 `"reactCompiler": true`만 추가하면 된다.  
memo, useMemo, useCallback을 사용하지 않아도 컴파일러가 자동으로 최적화를 해준다.  
의존성 지옥에 더이상 갇히지 않아도 된다.  
덩달아 눈을 혼란스럽게 하던 의존성 배열을 더 이상 코드수준에서 신경쓰지 않아도 된다.  
가독성, 유지보수성, 성능까지 챙길 수 있다니!  

### 6) Validation

zod를 여태 제대로 쓰진 못했던 것 같다.  
type/interface는 항상 직접 정의했었다.  
하지만 zod.infer<Type>과 zod.refine을 이용해서 타입과 밸리데이션을 한 번에 정의할 수 있다.  
무조건적으로 zod를 사용하는 건 오버 엔지니어링일 수 있으나,  
타입에 대한 검증과 유효성 검사가 중요한 참조형 타입이라면, zod에 익숙해질 필요가 있다고 본다.  
여기서 주의해야 할 점은 UI에 대한 FormSchema와 API에 대한 Schema를 분리해야 한다는 것이다.  
사용자 입력은 string으로 일반화 시키는게 좋다.  
다만 API의 payload에 포함할 데이터는 number일수도 있다.  
따라서 이런 부분은 명확하게 분리해서 검증을 실시 해야한다.  

### 7) State Management

마지막으로 상태관리는 서버의 상태와 UI의 상태를 분리해서 관리해야 한다. (너무 당연할지도...)  
소유권과 생명주기가 각각 다르다.  
서버의 상태는 잠시 스냅샷을 빌려와서 브라우저에서 조작하는 것 뿐이다.  
반면 UI상태는 브라우저 메모리에만 존재하며 사용자가 완전히 제어한다.  
이 둘을 섞으면 서버 데이터 동기 로직이 UI 로직을 오염시키게 된다.  
만약, Tanstack Query의 stale Time이나, gcTime을 zustand에 구현한다고 생각해보자...  
감도 안온다.  

상태 관리 독립의 4대 명분

1. 데이터의 신뢰성과 원천(Source of Truth)의 차이
2. 캐시 무효화(Invalidation) 전략의 복잡성
3. 선언적 동기화 vs 명령적 업데이트
4. 렌더링 최적화 및 관심사 분리

위에 어려운 키워드는 3번 정도 찾아보면, 머리에 담길 것 같다.  

프로젝트 초기 단계에 위와 같은 아키텍쳐를 적용하는 것은 오버 엔지니어링일 수 있다.  
하지만 프로젝트가 커지고, 팀원이 늘어나고, 코드의 복잡도가 올라간다면,  
위와 같은 아키텍쳐는 필수불가결한 요소가 될 것이다.  
이제 여기서 모니터링과 GEO / SEO를 덧붙인다면, 어느정도 완성형의 단일 프로젝트 아키텍쳐가 완성될 것 같다.  
모노 레포와 MFE까지, 추가로 BFF까지 붙여나가면, 기업용 프론트엔드 아키텍쳐가 완성될 것 같다.  
한 단계씩 내 지식을 쌓아보자.
