# Zod와 React Hook Form으로 만드는 안전한 폼

**"어떻게 하면 사용자 경험(UX)을 해치지 않으면서도 서버가 원하는 깨끗한 데이터를 보낼 것인가"**  

## 1. Zod vs TypeScript (런타임과 컴파일타임)
TypeScript는 컴파일 타임에만 작동하지만, **Zod**는 실제 실행 중(Runtime)에 들어오는 데이터를 검사한다.  

* **내가 오해했던 부분**: `z.number()`만 쓰면 문자열 `"25"`를 숫자로 알아서 바꿔줄 줄 알았다.
* **해결**: HTML `input`은 기본적으로 모든 값을 문자열로 반환한다. 이를 숫자로 다루려면 `z.coerce.number()`를 사용해 강제 형변환(Coercion)을 먼저 거쳐야 한다.

## 2. `flatMap` of JS 
Zod의 에러 메시지는 한 필드에 여러 개가 쌓일 수 있어 `string[]` 형태로 관리된다.  
이를 다룰 때 유용한 도구가 `flatMap`이다. 한꺼플 벗겨내보자. 

* **특징**: `map`으로 데이터를 가공한 뒤, 결과 배열을 1단계 평평하게(`flat`) 만든다.
* **활용**: 빈 배열 `[]`을 반환하여 특정 조건을 필터링하거나, 하나의 요소를 여러 개로 확장할 때 성능과 코드 가독성 면에서 유리하다.

## 3. React Hook Form (RHF)
RHF는 성능을 위해 **비제어 컴포넌트** 방식을 사용하며, `register`와 `handleSubmit`이 핵심이다.  

* **`register`**: 단순한 속성 부여가 아니라, 해당 입력창을 RHF 시스템에 등록하고 `onChange`, `onBlur`, `ref`를 자동으로 연결해 주는 브릿지 역할을 한다.  
* **`handleSubmit`**: 폼 제출 시 `e.preventDefault()`를 대신 호출하고, `zodResolver`를 통해 모든 데이터가 유효할 때만 최종 콜백을 실행하는 '가드(Guard)'이다.  

## 4. UX를 고려한 검증 모드 설정
에러 메시지가 갑자기 나타나 레이아웃이 흔들리는 현상(CLS)을 방지하기 위한 설정 도 있다.  

| 설정 항목 | 설명 | 권장 설정 |
| :--- | :--- | :--- |
| **`mode`** | 첫 유효성 검사 실행 시점 | `"onBlur"` (포커스 나갈 때) |
| **`reValidateMode`** | 에러 발생 후 재검사 시점 | `"onBlur"` (수정 후 나갈 때) |

**Tip**: 에러가 발생한 상태에서 글자를 입력하자마자 에러를 지워주고 싶다면 `reValidateMode`를 `"onChange"`(기본값)로 두는 것이 일반적!  
근데 CLS 걱정 된다.
