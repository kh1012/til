# 컴포넌트 갤러리 사용성 50라운드 마무리: 개인화 표면부터 테스트 스위트와 데이터 이식성까지

> 컴포넌트 갤러리 usability 라운드 16에서 50까지를 하루에 몰아친 날. 검색 복구와 팔레트 개편, 홈 개인화 레일, 상세 상호 연결, 키보드-온리 조작, 전역 토스트를 얹고, 순수 검색 로직을 추출해 44테스트 스위트를 세우고, 개인 데이터 백업/복원까지 붙여 50라운드를 닫았다.

## 배경

며칠간 ui-harness 컴포넌트 갤러리를 팀 실사용 단계까지 올려온 흐름의 꼬리였다. 이날은 사용성 개선을 라운드 단위로 세어가며 16번째부터 50번째까지, 그러니까 그동안 쌓은 개선 루틴을 한 번에 소진하며 "50라운드"라는 규율 목표를 닫는 날이었다.

작업의 결은 크게 다섯 갈래로 흘렀다. 첫째, 검색 실패에서 빠져나올 길과 자주 쓰는 것에 빨리 닿는 개인화 표면(팔레트 3구간, 홈 자주 본 레일)을 깔았다. 둘째, 상세 페이지를 이전/다음, 관련 컴포넌트, 쓰이는 곳, 브레드크럼으로 서로 연결해 탐색이 끊기지 않게 했다. 셋째, 그리드 로빙 내비와 p 키 핀 같은 키보드-온리 조작을 넓히고 단축키 도움말로 발견성을 챙겼다. 넷째, 순수 검색 헬퍼를 lib/search로 뽑아내고 그 위에 8파일 44테스트를 세워 회귀를 막았다. 다섯째, 핀·최근·조회기록을 JSON으로 내보내고 되불러오는 개인 데이터 이식성과 접근성 랜드마크로 마감했다. 하루 내내 한 라운드가 하나의 얇은 개선을 얹고 브라우저나 유닛 테스트로 닫는 리듬이 반복됐다.

## 핵심 내용

### 개별 작업 기록 (시간순)

- 결과 없음 복구: '다른 검색어로 다시' 최근 검색어 칩 (round 16)
  - 검색이 0건일 때 막다른 화면 대신, 최근 검색어를 칩으로 제시해 곧장 다른 질의로 갈아탈 길을 열었다. 방금 실패한 질의는 칩에서 제외해 같은 막다른 길로 되돌아가지 않게 했다.

- ⌘K 팔레트를 고정됨→최근 본→전체 3구간으로 (round 17)
  - 커맨드 팔레트를 단일 목록에서 고정됨, 최근 본, 전체 세 구간 헤더로 나눴다. 자주 쓰는 컴포넌트가 위로 올라와 팔레트가 개인화된 진입점이 되게 했다.

- 홈 '자주 본' 레일 (round 18)
  - IndexedDB에 쌓인 조회수 상위(2회 이상, 고정·최근과 중복 제외)를 홈 레일로 노출했다. 핀이 명시적 즐겨찾기라면 이건 빈도 기반 암묵적 즐겨찾기로, 손이 자주 가는 컴포넌트에 빠르게 닿게 한다.

- 검색 드롭다운 키보드 힌트 푸터 (round 19)
  - 검색 드롭다운 하단에 위아래 이동, 엔터 실행, esc 닫기 힌트를 노출했다. 키보드로 팔레트를 몰 수 있다는 사실 자체의 발견성을 올렸다.

- 갤러리 사용성·개인화 기능 노트 문서화 (round 20)
  - 지금까지 얹은 개인화 기능들의 저장소 키, 기능, 원칙을 usability 문서에 정리했다. 라운드가 늘면서 로컬 저장 스키마가 흩어지지 않도록 한곳에 모은 것이다.

- 상세 이전/다음 컴포넌트 이동 (round 21)
  - 목록 브라우징 순서를 기준으로 상세에서 좌우 화살표로 이웃 컴포넌트를 오갈 수 있게 했다. 목록으로 돌아갔다 다시 들어오는 왕복을 없앤다.

- 상세 관련 컴포넌트 (round 22)
  - 공유 키워드와 같은 카테고리를 근거로 관련 컴포넌트 상위 6개를 상세에 붙여 탐색을 옆으로 잇게 했다.

- 리스트 뷰 행에도 고정(핀) 버튼 (round 23)
  - 그리드 카드에만 있던 핀 버튼을 리스트 뷰 행에도 넣어 두 뷰의 조작 패리티를 맞췄다.

- 그리드 로빙 키보드 내비 (round 24)
  - 카드에 포커스가 있을 때 방향키로 이웃 카드로 이동하게 했다. 열 수를 실측해 위아래 이동이 실제 격자를 따르도록 계산했다.

- 카테고리 퀵필터 칩 (round 25)
  - 기본 화면에서 분류별로 좁혀 보는 칩을 넣고 토글·초기화와 연동했다. 검색어 없이도 분류만으로 범위를 좁히는 진입을 열었다.

- 갤러리 스크립트 재편 + 문서 포트 정정
  - ui:gallery를 ui:components-gallery로 바꾸고 ui:page-gallery를 신설했으며, 컴포넌트 스킬과 page-create를 함께 설치하고 stale을 정리하는 통합 skills 스크립트를 넣었다. README의 갤러리 포트 오류(:2026을 :9221로) 3곳도 바로잡았다.

- 전역 토스트 + 핀 액션 피드백 (round 26)
  - 커스텀 이벤트와 ToastHost로 전역 토스트를 세우고, 고정/해제 시 하단 토스트로 확인을 띄웠다. 이후 라운드의 공유·복사 피드백이 모두 이 토스트로 수렴한다.

- ⌘K 팔레트 인라인 핀 토글 (round 27)
  - 팔레트를 떠나지 않고 그 자리에서 핀을 토글하게 했다. 고정하려고 상세로 들어갔다 나오는 흐름을 없앤다.

- 상세 페이지 고정 토글 (round 28)
  - 상세를 보는 중에 바로 핀을 걸 수 있게 하고, 토스트 확인과 홈 고정됨 레일 연동까지 이었다.

- 카테고리 필터 URL 동기화(?cat=) (round 29)
  - 카테고리 필터를 쿼리스트링에 실어 딥링크와 새로고침 복원이 되게 했다. 기존 filter/sort/q 동기화와 일관된 방식이다.

- 상세 브레드크럼 (round 30)
  - 상세 상단에 라이브러리 › 카테고리 브레드크럼을 넣고, 카테고리를 누르면 그 분류로 필터된 목록으로 딥링크되게 했다.

- 상세 import 문 복사 버튼 (round 31)
  - 상세에서 해당 컴포넌트의 import 구문을 한 번에 복사하는 버튼을 넣었다. 갤러리에서 코드로 옮겨 붙이는 마찰을 줄인다.

- 포커스 카드에서 p 키로 핀 토글 (round 32)
  - 그리드 키보드 내비를 연장해, 포커스된 카드에서 p 키로 핀을 토글하게 했다. 마우스 없이 고정까지 마치는 키보드-온리 경로가 완성됐다.

- 단축키 도움말에 라이브러리·상세 섹션 추가 (round 33)
  - 새로 늘어난 그리드 내비, p 핀, 상세 좌우 이동 같은 키보드 기능을 단축키 도움말에 섹션으로 노출해 발견성을 챙겼다.

- 공유 링크 복사 시 토스트 확인 (round 34)
  - 링크 복사에도 '링크를 복사했어요' 토스트를 붙여 피드백을 일관되게 맞췄다.

- 상세 공유 링크 버튼 (round 35)
  - 상세에서 그 컴포넌트로의 딥링크를 복사하는 공유 버튼을 넣고, 위의 토스트 확인과 연결했다.

- 순수 검색 헬퍼를 lib/search.ts로 추출 (round 36)
  - LibraryRoute 안에 얽혀 있던 검색 로직(부분수열 매칭, score 등)을 순수 함수로 lib/search.ts에 뽑아냈다. 라우트를 슬림하게 만드는 동시에, 다음 라운드에서 테스트 가능하게 만들기 위한 준비였다(동작 보존).

- lib/search 유닛 테스트 15개 (round 37)
  - 방금 추출한 검색 헬퍼의 isSubsequence, matches, score를 15개 테스트로 덮어 회귀를 막았다. 추출과 테스트를 연달아 붙이는 리듬의 시작점이다.

- recentSearches 유닛 테스트 7개 (round 38)
  - 최근 검색어의 중복 제거, 상한, 정규화, 삭제를 메모리 localStorage 스텁 위에서 7개 테스트로 검증했다.

- pins 유닛 테스트 5개 (round 39)
  - 핀 토글, 고정 여부 조회, 그리고 깨진 저장값에 대한 방어를 5개 테스트로 덮었다. 로컬 저장 데이터가 손상돼도 무너지지 않게 하는 게 핵심이었다.

- npm test 스크립트 추가 (round 40)
  - vitest run으로 gallery/src를 도는 test 스크립트를 넣어, 그때까지의 3파일 27테스트가 한 명령으로 발견·실행되게 했다.

- usage 유닛 테스트 4개 (round 41)
  - IndexedDB를 모킹해 조회수 기록의 count 증가와 동시 호출 직렬화를 4개 테스트로 검증했다.

- idb 안전 무력화 테스트 3개 (round 42)
  - IndexedDB가 없는 환경에서 조회 기록이 undefined 또는 no-op로 안전하게 죽는지 3개 테스트로 확인했다(전체 34테스트).

- 개인 데이터 내보내기/가져오기 (round 43)
  - 핀·최근·검색·설정·조회기록을 JSON으로 백업하고 되불러오는 기능을 3개 테스트와 함께 넣었다. 라운드 내내 쌓아온 로컬 개인화가 브라우저에 갇히지 않고 이식되도록 하는 출구다.

- 카드 요약에도 검색어 하이라이트 (round 44)
  - 이름뿐 아니라 설명에 걸린 매칭도 하이라이트해, 왜 이 카드가 검색에 걸렸는지를 시각적으로 드러냈다.

- 상세 '쓰이는 곳'(used-by) (round 45)
  - 이 컴포넌트를 composedOf로 쓰는 상위 컴포넌트를 역링크로 걸었다. 관련 컴포넌트가 옆으로 잇는다면 used-by는 위로 잇는 탐색이다.

- toast 유닛 테스트 2개 (round 46)
  - 전역 토스트의 이벤트 타입과 detail, 그리고 window가 없는 환경에서의 안전을 2개 테스트로 덮었다.

- 개인화 영역 랜드마크 a11y (round 47)
  - 카테고리를 group으로, 최근/자주 본을 라벨된 region으로 만들어 스크린리더 랜드마크를 붙였다. toast 테스트의 타입도 함께 고쳤다.

- USABILITY.md 전면 갱신 (round 48)
  - 라운드 1에서 47까지의 기능, 저장소 키, 테스트 스위트, 반응형 백로그를 문서에 전면 반영했다.

- recents 유닛 테스트 5개 + 전체 검증 스윕 (round 49)
  - 최근 본 목록 테스트 5개를 더해 8파일 44테스트로 늘리고, typecheck 0과 브라우저 스모크까지 전체 검증 스윕을 돌렸다.

- test:watch 스크립트 + 50라운드 마무리 (round 50)
  - 감시 모드 test:watch 스크립트를 넣고, 8파일 44테스트·typecheck 0으로 50라운드 사용성 강화를 닫았다.

- 신규 배너 아이콘 Sparkles → PackagePlus
  - 신규 컴포넌트 배너 아이콘이 별표(Sparkles)라 AI를 연상시켰다. 별표 AI 아이콘을 지양하는 방침에 맞춰 PackagePlus로 교체했다.

## 정리

이날 서른일곱 개 커밋을 하나로 꿰는 건 "컴포넌트 갤러리 사용성을 50라운드로 닫는다"는 규율이었다. 라운드를 세는 방식이 단순한 카운터가 아니라, 한 라운드에 얇은 개선 하나만 담고 반드시 브라우저나 유닛 테스트로 닫는 계약처럼 작동했다. 오후 초반에는 검색 실패에서 빠져나올 길과 개인화 진입점(팔레트 3구간, 자주 본 레일)을 깔고, 중반에는 상세를 이전/다음·관련·used-by·브레드크럼으로 서로 잇고 키보드-온리 조작을 넓혔으며, 후반에는 검색 로직을 추출해 테스트 스위트를 세우고 개인 데이터 이식성으로 마감했다.

가장 기억에 남는 건 라운드 36에서 39로 이어지는 추출-후-테스트 리듬이다. 검색 로직을 lib/search로 뽑아낸 이유가 "라우트를 슬림하게"인 동시에 "테스트 가능하게"였고, 실제로 바로 다음 라운드에서 15개 테스트가 그 위에 얹혔다. 순수 함수로 경계를 그으면 곧장 회귀 방어를 세울 수 있다는 걸, 추출과 테스트를 붙여 커밋하는 순서로 몸에 익힌 셈이다. pins·usage·idb 테스트가 하나같이 "깨진 저장값", "IndexedDB 부재" 같은 실패 경로를 먼저 덮은 것도 같은 맥락이다. 개인화를 로컬 저장에 얹는 순간 데이터 손상과 환경 부재가 1급 리스크가 되므로, 기능보다 방어를 먼저 테스트한 판단이 옳았다.

또 하나 남는 건 개인화가 처음엔 흩어진 편의 기능들이었다가 라운드가 쌓이며 하나의 계약으로 수렴했다는 점이다. 핀·최근·조회기록이 각자 로컬 저장 키로 흩어져 있던 걸 usability 문서로 한곳에 모으고(round 20, 48), a11y 랜드마크로 스크린리더에 노출하고(round 47), 끝내 JSON 백업/복원으로 브라우저 밖으로 내보낼 수 있게(round 43) 만들었다. 편의 기능을 계속 얹다 보면 저장 스키마가 부채가 되는데, 문서화-접근성-이식성으로 그 부채를 닫으며 50라운드를 마무리한 게 이날 사용성 작업의 실질적 성과였다.
