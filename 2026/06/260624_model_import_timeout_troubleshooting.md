# 모델 데이터 임포트 타임아웃, 네트워크 기록으로 좁히고 일단 풀기

> 큰 모델을 임포트할 때 데이터가 다 오기 전에 멈추는 증상이 있었다. 무엇이 막는지부터 잡으려고 model import 스트림 요청을 HAR로 떠서 트러블슈팅 디렉토리에 남겼고, 범인으로 지목된 부재력/단면 테이블 fetch의 120초 타임아웃을 우선 None으로 풀어 흐름을 통과시켰다. 근본 해결이 아니라 막힌 길을 임시로 뚫은 조치다.

## 배경

design-core의 모델 임포트는 maxapi에서 모델 데이터를 받아오는데, 그 안에서 부재력 테이블(force table)과 단면 테이블(section table)을 httpx로 따로 가져오는 단계가 있다. 두 fetch 모두 응답이 120초 안에 와야 한다는 타임아웃이 걸려 있었다.

문제는 모델이 커지면 이 응답이 120초를 넘긴다는 점이었다. 임포트가 끝까지 가지 못하고 중간에 끊겼는데, 끊기는 지점이 클라이언트 표면에서는 분명하게 드러나지 않았다. 그래서 먼저 "어느 요청에서, 어떻게 멈추는가"를 눈으로 확인할 수 있게 만드는 일부터 시작했다.

## 핵심 내용

### 개별 작업 기록 (시간순)

- model import 스트림 요청을 HAR로 캡처해 트러블슈팅 기록으로 남김
  - 임포트가 막히는 지점을 좁히려고 `/maxapi/v1/model/import/stream` 요청의 네트워크 기록을 HAR로 떠서 `packages/design-core/.troubleshooting/260624-model-data-import-error/` 아래에 저장했다. 증상을 말로만 적어두는 대신, 실제로 오간 요청을 그대로 박제해 둔 것이다. 날짜와 증상 이름을 붙인 트러블슈팅 폴더를 두는 방식이라, 나중에 같은 증상을 다시 만났을 때 그때의 네트워크 흐름을 곧장 대조할 수 있게 했다.

- 부재력/단면 테이블 fetch의 타임아웃을 120초에서 None으로 임시 제거
  - 네트워크 기록으로 막히는 자리를 좁힌 뒤, `maxapi_model_import_service.py`의 `_fetch_force_table`과 `_fetch_section_table` 두 곳에서 httpx 요청의 `timeout=120.0`을 `timeout=None`으로 바꿨다. 큰 모델에서 응답이 120초를 넘겨 타임아웃 예외로 떨어지던 흐름을, 일단 끝까지 기다리게 풀어 임포트가 완료되도록 통과시켰다. 커밋 메시지에도 "임시 제거"라고 적었듯, 이건 막힌 길을 우선 뚫은 조치지 응답이 느린 원인 자체를 고친 것은 아니다.

## 정리

이날 백엔드 쪽은 "막히는 증상을 어떻게 좁히고, 일단 어떻게 통과시키는가"의 짧은 한 사이클이었다. 임포트가 도중에 끊긴다는 모호한 증상을 그대로 두지 않고, model import 스트림 요청을 HAR로 떠서 실제 네트워크 흐름을 손에 쥐는 일부터 했다. 증상을 재현 가능한 자료로 바꿔두면 추측이 줄고, 다음에 같은 자리에서 막혀도 대조할 기준점이 생긴다. 트러블슈팅 폴더를 날짜와 증상 이름으로 남긴 것도 같은 의도다.

타임아웃을 None으로 푼 것은 정직하게 말해 임시 조치다. timeout=None은 응답이 영영 안 와도 무한정 기다린다는 뜻이라, 큰 모델에서 임포트가 통과되는 대신 다른 상황에서는 요청이 매달려 버릴 위험을 안고 있다. 진짜 해결은 둘 중 하나일 것이다. maxapi가 부재력/단면 테이블을 더 빨리 내려주게 만들거나, 한 번에 다 받지 않고 스트리밍이나 페이지 단위로 잘라 받아 각 청크가 합리적인 타임아웃 안에 들어오게 하는 것. 오늘은 거기까지 가지 않고, 우선 막힌 임포트를 통과시켜 작업이 멈추지 않게 하는 선에서 끊었다. 임시라는 사실을 코드와 기록 양쪽에 남겨 둔 만큼, 근본 해결은 별도 과제로 남겨진 셈이다.
