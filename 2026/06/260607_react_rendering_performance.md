# React 렌더링 성능 & 호가창 최적화 — 한 프레임 예산으로 모인다

> 모든 최적화는 결국 하나다 — 메인 스레드의 '한 프레임 예산'을 지킨다. 줄이거나, 미루거나, 옆 차선으로 옮긴다. 무엇을 적용할지는 항상 '측정'으로 정한다.

## 배경

실시간 호가창(orderbook)처럼 초당 수십 번 갱신되는 UI를 다루다 보면, "리렌더가 많다 = 느리다"라는 직관만으로는 어디를 고쳐야 할지 알 수 없다. React 렌더링이 실제로 어떤 단계를 거치고, 무엇이 비용이고, 그 비용이 어디(스레드)에서 경쟁하는지를 정리하면 최적화 도구를 '감'이 아니라 '근거'로 고를 수 있다. 학습한 6개 개념이 결국 하나의 원리(한 프레임 예산)로 수렴하는 흐름으로 정리한다.

## 핵심 내용

### 1. 렌더링은 두 단계다

- **렌더 단계**: 컴포넌트 함수를 다시 호출해서 새 화면 구조(가상 DOM)를 계산한다.
- **커밋 단계**: 이전 구조와 비교(diff)해서 진짜 바뀐 부분만 실제 DOM에 반영한다.

> 핵심 — 함수가 100번 다시 불려도 실제 화면은 0번 바뀔 수 있다.

"리렌더 = 화면 갱신"이 아니다. 함수 재호출(렌더)과 DOM 변경(커밋)은 별개다. 그래서 '리렌더 횟수'만 보고 패닉할 필요는 없다. 진짜 비용이 어디서 나는지를 봐야 한다.

### 2. 언제 다시 그려지나 — 3가지 계기

1. 내 **상태(state)** 가 바뀐다
2. 받는 **props** 가 바뀐다
3. **부모가 다시 그려진다**

3번이 가장 많이 놓치는 함정이다:

> 부모가 리렌더되면, 자식은 props가 안 바뀌어도 전부 따라 그려진다.

```
        부모 리렌더
       /     |     \
    자식    자식    자식   ← props 그대로여도 전부 재호출
```

### 3. 그래서 왜 문제인가

호가창 기준 체감:

```
50행 × 초당 30회 = 초당 1,500 렌더
```

렌더 부하는 **두 종류**로 나뉜다.

- **비교 비용** — 리액트가 새 구조를 만들고 diff 하는 비용. (개수 × 빈도로 누적)
- **계산 비용** — 렌더 함수에 넣은 무거운 계산. (정렬·누적합 등)

> 진단이 먼저 — 둘 중 무엇이 병목인지 반드시 측정으로 구분한다.

비교 비용이 병목이면 `React.memo`/구조 분리가 답이고, 계산 비용이 병목이면 `useMemo`/워커가 답이다. 진단 없이 도구를 바르면 엉뚱한 곳을 고치게 된다.

### 4. 어디서 실행되나 — 실행 환경

```
한 프레임 예산 = 1000 ÷ 주사율   (반비례)
60Hz → 16.67ms · 120Hz → 8.3ms · 30Hz → 33.3ms
```

- 자바스크립트는 **단일 메인 스레드**에서 실행된다. (한 번에 한 줄 — 단일 차선)
- **비교 비용·계산 비용 모두 같은 메인 스레드 위에서 경쟁**한다.
- 예산을 넘기면 스타일·레이아웃·페인트가 밀려 **프레임 누락 = 버벅임**.
- **GPU = 합성·WebGL 그리기 담당 별도 차선**.

이게 모든 최적화의 기준점이다. "왜 16.67ms 안에 끝내야 하나?" → 60Hz 화면에서 한 프레임에 주어진 시간이 딱 그만큼이기 때문이다. 이 예산을 넘기는 순간 프레임이 떨어진다.

### 5. 실행 순서와 우선순위

- **한 렌더 패스 내부**: 컴포넌트는 트리 순서로 순차 실행되고 비용은 합산된다. 재정렬·병렬 없음 → **순서가 아니라 '총합'이 문제**.
- **우선순위는 컴포넌트가 아니라 '업데이트 단위'에 있다** (Concurrent):
  - 급한 갱신(입력·클릭)은 먼저
  - 미뤄도 되는 갱신은 쪼개서 양보

> 쪼개서 양보하는 도구 → `useTransition` · `useDeferredValue`

Concurrent의 핵심은 "어떤 컴포넌트가 우선이냐"가 아니라 "어떤 업데이트가 급하냐"다. 입력 응답은 즉시, 그에 따른 무거운 리스트 갱신은 양보(deferred)시켜 메인 스레드를 양보한다.

### 6. 최적화 도구와 기준

- **`React.memo`** — props 안 바뀐 자식의 '따라 그리기'를 끊는다. (계기 ③ 차단)
- **`useCallback`** — 함수가 매번 새로 태어나 memo가 깨지는 걸 막는다. (memo와 짝)
- **`useMemo`** — 무거운 계산 결과를 기억해 반복 계산을 막는다. (계산 비용 차단)

적용 기준:

```
절약량 = 빈도 × 한 번 비용   (둘 다 클 때만 바르는 게 이득)
```

- **측정 먼저(Profiler) → 병목에만 적용** (다 바르면 오히려 손해)
- 더 무거우면 차선을 옮긴다 — **웹 워커**(순수 계산을 별도 스레드)
- 렌더 규모별 기술 선택:

```
DOM (수백) → Canvas (수천~수만) → WebGL (수십만 · GPU)
```

`memo`/`useMemo`도 공짜가 아니다(비교·캐시 비용이 든다). 빈도와 한 번 비용이 둘 다 클 때만 절약량이 오버헤드를 넘는다. 그래서 "전부 바르기"는 안티패턴이고, Profiler로 병목을 짚은 뒤 그 지점에만 발라야 한다.

## 정리

핵심 원리 하나로 모인다:

> **모든 최적화는 결국 메인 스레드의 '한 프레임 예산'을 지키는 것이다.**

이 예산을 지키는 방법은 세 갈래뿐이다.

1. **줄인다** — `React.memo` · `useMemo` · `useCallback` (불필요한 렌더/계산 제거)
2. **미룬다** — `useTransition` · `useDeferredValue` (급한 것 먼저, 나머지 양보)
3. **옆 차선으로 옮긴다** — 웹 워커(계산) · Canvas/WebGL·GPU(그리기)

그리고 무엇을 적용할지는 **항상 측정으로 정한다.** 비교 비용이 병목인지 계산 비용이 병목인지부터 Profiler로 진단하고, 절약량(빈도 × 한 번 비용)이 도구의 오버헤드를 넘을 때만 바른다.

오늘 가장 크게 바뀐 관점은 두 가지다.

- 리렌더 횟수 자체가 죄가 아니다. **렌더(함수 호출)와 커밋(DOM 변경)은 별개**이고, 진짜 비용은 비교 비용 / 계산 비용으로 나뉜다.
- 모든 도구(memo / transition / worker / WebGL)는 서로 다른 트릭이 아니라 **'한 프레임 예산'이라는 같은 목표를 다른 차선에서 푸는 방법**이다. 그래서 도구 선택의 기준이 "유행"이 아니라 "어느 차선이 막혔는가(측정)"로 통일된다.
