# MCP URL을 사파리로 열어보다 만난 JSON-RPC와 Transport

> 표면의 에러 메시지 한 줄을 따라 내려갔더니, 결국 MCP라는 추상화의 뼈대가 보였다.

## 배경

GPT 앱 빌더에 MCP 서버를 등록하려고 Cloudflare Tunnel로 로컬 서버를 외부에 노출하던 중이었다. URL이 제대로 살아있는지 궁금해서 별 생각 없이 사파리 주소창에 그 URL을 그대로 붙여넣고 엔터를 쳤더니, 화면에 떡하니 JSON-RPC 에러가 떠 있었다. 처음에는 "어, 뭐가 잘못된 거지" 싶다가, 메시지를 한 번 더 읽어보니 오히려 묘하게 안심이 됐고, 그 안심에서 한 발짝 더 들어가 보니 그동안 내가 RPC라는 멘탈 모델을 별로 깊이 들여다본 적이 없었다는 사실을 마주하게 됐다.

## 핵심 내용

### 사파리에 떠 있던 그 한 줄

화면에 찍힌 응답은 이런 모양이었다.

```json
{
  "jsonrpc": "2.0",
  "id": "server-error",
  "error": {
    "code": -32600,
    "message": "Not Acceptable: Client must accept text/event-stream"
  }
}
```

처음 든 생각은 "MCP URL이 깨졌나" 였는데, 메시지를 천천히 읽고 나니 거꾸로 보였다. 클라이언트가 `text/event-stream`을 받아들일 수 있어야 한다고 서버가 또박또박 말해주고 있었으니까, 이건 서버가 죽은 게 아니라 멀쩡히 살아서 정중하게 거절한 응답이었다. 사파리는 그저 평범한 브라우저답게 `Accept: text/html`로 GET을 보냈고, MCP의 Streamable HTTP transport는 `Accept: application/json, text/event-stream`을 요구했으니, 이 둘이 맞지 않아 거절당한 게 전부였다. 에러가 떴다는 사실 자체가 오히려 서버가 정상적으로 동작 중이라는 가장 확실한 신호였던 셈이다.

### MCP URL이라는 게 결국 뭔가

MCP URL은 이름이 거창해 보였지만 따져보면 그냥 MCP 서버의 HTTP 엔드포인트였다. 내 노트북 안에서 돌아가는 MCP 서버가 있고, Cloudflare Tunnel이 그걸 공인된 도메인 하나에 매핑해주고, GPT나 Claude Desktop 같은 클라이언트가 그 URL로 JSON-RPC 메시지를 POST해서 tools나 resources, prompts를 호출하는 구조였다. 그러니까 "MCP URL을 등록했다"는 말은, 내 로컬 도구들을 외부 LLM 클라이언트가 함수처럼 부를 수 있는 입구를 하나 열어줬다는 뜻이었다.

### JSON-RPC라는 형식

여기서 멈췄으면 됐을 텐데, 응답 본문에 박혀 있던 `"jsonrpc": "2.0"`이 자꾸 눈에 밟혔다. JSON-RPC가 뭔지 머릿속으로는 안다고 생각했지만, 막상 한 줄로 설명해보라고 하면 어물거릴 것 같았다. 정리해보면 JSON-RPC는 JSON 포맷으로 원격 함수를 호출하는 약속이고, 핵심은 "함수 호출"이라는 멘탈 모델 쪽에 있었다. REST에 익숙해져 있다 보니 이게 묘하게 낯설었다. REST는 리소스를 중심에 두고 GET, POST, PUT, DELETE 같은 동사로 그 리소스에 무언가를 한다는 식인데, JSON-RPC는 그런 리소스 추상화를 거의 들이지 않고, 그냥 "이 함수를 이 인자로 불러달라"고 직접 말한다.

요청은 대략 이런 모양이고,

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": { "name": "search", "arguments": { "q": "mcp" } }
}
```

응답은 성공이든 실패든 같은 봉투에 담겨 돌아온다.

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { "content": [...] }
}
```

아까 봤던 `-32600`은 JSON-RPC 표준 에러 코드 중 Invalid Request에 해당하는 값으로, "메시지 자체가 사양을 벗어났다"는 뜻이었다. 사파리의 GET이 JSON-RPC 입장에서는 애초에 말이 안 통하는 형식이었으니 합리적인 응답이긴 했다.

### RPC라는 오래된 아이디어

JSON-RPC를 보다 보니 자연스럽게 한 발 더 내려가 RPC 자체가 뭔가에 대한 질문이 생겼다. RPC, Remote Procedure Call. 풀어쓰면 "원격 프로시저 호출"인데, 이름이 사실상 정의를 다 담고 있다. 네트워크 너머에 있는 함수를, 마치 내 코드 안에 정의된 함수인 것처럼 부르고 싶다는 욕망이 그 출발점이다.

```ts
const result = sum(1, 2);
```

이렇게 부르나,

```ts
const result = await rpc.call('sum', [1, 2]);
```

이렇게 부르나, 호출자의 머릿속에서는 "함수를 부른다"는 감각이 똑같이 유지되도록 설계하려는 시도, 그게 RPC다. JSON-RPC는 그 메시지를 JSON으로 표현했을 뿐이고, gRPC는 같은 가족 안에서 메시지를 Protobuf로 표현한 형제 같은 친구다. REST에 너무 오래 머물러 있다 보니 이런 함수 중심 멘탈 모델이 한동안 어색했는데, MCP처럼 "tools를 호출한다"는 의미가 강한 도메인에서는 RPC 쪽이 더 자연스럽게 들어맞는다는 생각이 그제서야 들었다.

### 그렇다면 transport는 또 뭔가

여기까지 와서 마지막으로 남은 단어가 transport였다. MCP 문서를 처음 볼 때부터 "stdio transport"니 "HTTP transport"니 하는 표현이 자주 나왔는데, 이걸 그동안은 그냥 "통신 방식"이라고 뭉뚱그려 이해하고 있었다. 근데 왜 굳이 transport를 따로 분리해서 부를까, 라는 질문을 한 번 던져보니, 의외로 깔끔한 그림이 그려졌다.

메시지가 어떻게 생겼는지를 정의하는 것과, 그 메시지를 어떤 수단으로 전달할지를 정의하는 것은 사실 별개의 관심사다. 편지에 비유하자면, 편지지 위에 어떤 형식으로 글을 쓸지가 프로토콜이고, 그 편지를 우체국으로 보낼지 비둘기로 보낼지 손으로 직접 건넬지가 transport에 해당한다. 같은 편지를 여러 방식으로 보낼 수 있고, 같은 운송 수단으로 다양한 종류의 편지를 보낼 수 있다는 그 직교성을 살리려고 두 층위를 분리해 둔 거였다.

MCP는 이 자리에서 두 가지 transport를 정의해 둔다. 하나는 stdio transport로, 클라이언트가 MCP 서버를 자식 프로세스로 띄우고 그 프로세스의 stdin과 stdout을 메시지 통로로 쓰는 방식이다. 같은 머신 안에서 돌아갈 때는 네트워크 스택을 거치지 않으니 빠르고, 인증이나 네트워크 노출을 신경 쓸 필요도 거의 없다. 다른 하나가 이번에 사파리로 두드려본 HTTP transport, 정확히는 Streamable HTTP다. 클라이언트가 POST로 요청을 보내고, 서버는 응답을 SSE 스트림으로 흘려보낸다. 왜 굳이 SSE까지 써야 했는가 하면, 단순히 한 번 요청하고 한 번 응답받는 모델로는 표현하기 어려운 상황들이 MCP에는 흔하기 때문이다. 도구 실행이 길어지는 동안 진행 상황을 보내주거나, 서버가 클라이언트 쪽 LLM에게 거꾸로 sampling을 요청하는 식의 양방향 흐름이 필요하니까, 그 역방향 채널을 깔아두기 위해 SSE가 자연스럽게 따라 붙는다. 그러니 `Accept: text/event-stream`이 필수였다는 점도, 사파리가 거절당한 이유도, 이 한 층 더 내려와서야 비로소 "그렇게 설계할 수밖에 없었구나" 하고 납득이 됐다.

### 한 줄로 묶어보면

```
MCP = JSON-RPC 메시지를 stdio 또는 HTTP transport로 주고받는 프로토콜
```

이렇게 한 줄로 적어 두고 보니, 처음 사파리에 떴던 그 짧은 에러 메시지가 사실은 이 모든 층위를 압축해서 보여주고 있었다는 생각이 든다. JSON-RPC라는 메시지 형식, transport라는 전달 계층, 그리고 Streamable HTTP가 SSE를 요구하는 이유까지, 그 한 줄 안에 다 들어 있었던 셈이다.

## 정리

처음에는 그냥 "URL이 살아있는지 확인하고 싶었던" 정도의 가벼운 호기심이었는데, 에러 한 줄을 무시하지 않고 따라 내려가보니 MCP의 설계 의도까지 닿았다. 평소에 REST 기반 API를 다루면서 굳어 있던 멘탈 모델이 MCP를 만나면서 한 번 흔들렸고, RPC라는 함수 호출 중심의 사고방식이 어느 도메인에서는 훨씬 더 자연스럽게 들어맞을 수 있다는 걸 새삼 깨달았다. 그리고 메시지 형식과 전달 수단을 분리해서 사고하는 습관도, 이번처럼 transport라는 말을 두 번 세 번 곱씹어 본 다음에야 손에 잡히는 감각으로 바뀌었다.

표면에서 만난 에러를 그냥 닫아버리지 않고 한 단계만 더 따라 들어가보면, 그 안에 숨어 있던 설계자의 결정과 의도가 슬쩍 드러나는 순간이 있다. 오늘 사파리 주소창에서 시작된 이 짧은 산책이 딱 그런 경험이었고, 다음에 또 다른 낯선 에러를 만났을 때도 비슷한 호기심으로 한 번 더 내려가볼 수 있을 것 같다.
