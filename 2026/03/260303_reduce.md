# Array Reduce

> reduce는 시간복잡도가 O(n)인 Array의 메서드로,  
> 메서드명에서도 알 수 있듯이, 배열을 압축할 수 있고 변환할 수 있으며, 새로운 배열이나 값을 생성하며 불변성을 유지할 수 있게 해준다.

만약 `배열에 주어진 숫자를 같은 숫자 단위로 블록화 하고 싶다`면 어떻게 해야할까?  
for...of문을 사용해서 같으면 push하고 다르면 새로운 배열을 생성하여 push하는 방식으로 구현할 수 있다.  
이게 오히려 성능상 가장 효율적일 순 있다.  
하지만 reduce를 사용한다면 코드가 더 간결해질 뿐만아니라, 가독성도 훌륭해지는 거 같다.  
```ts
const toBlocks = (arr: number[]) => arr.reduce((blocks, item) => {
  //핵심은 블록화 할 2차원 배열의 마지막 요소인 1차원 배열을 꺼내와서,
  //현재 요소와 같은 숫자라면 추가 해주고, 다른 숫자라면 새로운 1차원 배열을 생성해주면 된다.
  const lastBlock = blocks[blocks.length - 1];
  if (lastBlock && lastBlock[0] === item) {
    lastBlock.push(item);
  } else {
    blocks.push([ item ]);
  }
  return blocks;
}, []);
```

그럼 이번에는 `블록화된 2차원 배열을 짝과 홀을 구분하여 2개씩 합친 배열을 만들고 싶다`면 어떻게 해야할까?  
예를 들어,
```js
const matrix = [
  [0],
  [1],
  [0],
  [1], //홀, 짝 2개씩, 반복 구간
  [0],
  [1]
];

const result = [
  [0, 0],
  [1, 1],
  [0, null],
  [1, null]
]
```
2차원 배열(matrix)을 짝과 홀을 구분하여 2개씩 합친 배열을 만들고 싶다!  
역시 또 reduce를 통해 변환 하면 된다.  
그리고 여기서는 flat([,depth]) 메서드도 살짝 한방울 떨어뜨린다.  
아주 중요한 조건이 있다.  
짝과 홀을 구분해야 하고 2개씩 합쳐야 하므로, 요소 4개 당 반복이 발생된다.  
익히 잘 알고 있는 i % 4를 쓰면 0, 1, 2, 3이 반복되고,
i + 2를 하면 `[matrix[0], matrix[2]], [matrix[1], matrix[3]]`이 되어  
다음 짝/홀 인덱스의 값과 쌍을 이룰 수 있게 된다.  
그리고 flat([,depth]) 메서드를 사용하여 2차원 배열을 1차원 배열로 변환한다.  
```ts
const toPairedBlocks = (matrix: number[][]) => matrix.reduce((blocks, arr, idx) => {
  //4개 단위로 끊고, 처음 2개 인덱스 일 때, 다음 짝/홀 값과 짝지어 배열을 만들어 준다.
  if ((i % 4) < 2) {
    const nextArr = matrix[idx + 2] ?? null;
    blocks.push([arr, nextArr].flat()); //범위를 벗어나 값이 존재하지 않는다면 null로 대체한다.
  }
  return blocks;
}, []);
```

마지막으로 `2차원 배열의 각 요소 길이 중, 최대 길이를 구해본다`면?  
불필요한 메모리 공간 차지일 수 있으나, 코드 일관성과 디버깅 시 일관성을 위해 null을 사용한 부분을 고려하여 필터링 해야한다.  
```ts
const toMaxLength = (matrix) => matrix.reduce((maxLength, arr) => {
  const length = arr.reduce((count, item) => item !== null ? count + 1 : count, 0);
  return Math.max(maxLength, length);
}, 0);
```
처음에는 filter 메서드를 통해 접근했으나,  
공간 복잡도가 증가하여, O(N^2)이 될 우려가 있어, reduce로 변경하였다.  
제일 빠른 건 for...of문을 사용하는 것 이겠지만,  
미세한 성능 차이에 대한 트레이드 오프는 모던함을 통한 가독성에 투자하는게 맞지 않을까 싶다.  
  
어쩌다 보니, 수학적 논리와 사고를 프로그래밍 언어로 글을 작성하는 법을 익히게 되었다.  
사실 프론트앤드 개발에서는 거의 뭐.. 무지성 `map`과 `for` 그리고 `filter`정도를 사용하다 보니,  
자바스크립트라는 언어를 이해하지 못하고, 불필요한 코드가 너무나도 많았던게 아닐까라는 생각이 들었다.  
아무리 AI가 코드를 작성해준다 하더라도,  
아직은 에이전트 수준이기 때문에 결국 사람은 관리자 혹은 검토자 역할을 수행해야 한다.  
그리고 마이크로한 유지보수는 결국 사람이 수행할 수 밖에 없다.  
이 관점으로 미루어 볼 때, 기본을 단단히 다지는 건, 과거나 현재나 중요함이 틀림 없어 보인다.
