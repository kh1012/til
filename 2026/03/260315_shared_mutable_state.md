# Shared Mutable State

> 클로드 코드를 쓰면서, 모래성을 쌓는다는 느낌을 받았는데,  
> 결국 기록을 남겨가면서 설계자에서 감독관이 되어 가는게 가장 중요하다고 느껴진다.  

[참고영상 링크 @youtube](https://m.youtube.com/watch?v=6Z6Le3Xwqdg)  

## 핵심 원칙
- 기획과 코딩의 분리  
- 작성된 계획을 직접 검토하고, 승인하기 전까지 AI가 코드를 작성하지 못하게 하는 것.    
> 넌지시 맡겨보고 싶어서 초안을 작성해달라고 했던 나를 다시 돌아보게 되었다.  

## 리서치
- 코드 베이스를 먼저 읽게하려면, `research.md`를 작성하게 해라.  
- `상세하게`, `매우 상세히`등과 같은 키워드를 이용해서 겉핥기식으로 분석하는 걸 방지해라.
- 기존 시스템 파괴, 중복 코딩등과 같은 값 비싼 실패를 막기 위함.    
> 항상 코드베이스를 벗어나지 않도록 가이드 했었는데,   
> 이런 키워드는 하나씩 쌓아서 **md파일로 관리**해두는게 좋겠다.   
> 마치 스니펫처럼 각 단계별 지침서를 정의해두어야 겠다.

## 계획 수립 및 반복 검토 루프
- 별도 `plan.md` 파일에 구현 계획, 코드 스니펫, 수정 파일 경로, 고려 사항 등을 작성하게 해라.  
- 개발자가 직접 에디터에서 `plan.md`를 열어 직접 주석(메모)를 달며 잘못된 가정을 수정하거나 제약조건을 반드시 추가한다.  
- 아직 구현하지마 라는 명시적인 제안이 있어야, AI가 직접 수정하지 않는다.  
> 그저 읽어보고 수정만 했지, 직접 하나하나 수정하진 않았었는데...  
> 이 계획을 날짜와 시간 기준으로 추가해가면,  
> 나중에 문제를 찾거나 로깅하는데에도 도움이 될듯하네.

## 구현 (기계적 실행)
- AI가 지루한 코딩을 하게 해야 한다.     
- 구현 중에는 설계자에서 감독자로 역할을 변경하여, 타입 체크를 돌리거나 세부적인 디자인 수정을 지시한다. (반복적으로)  
> 사실, 은연중에 창의적으로 내가 생각해내지 못한 대단히 새로운 걸,
> 당연히 AI라면 찾아내서 개발 해줄거라 기대하고 있었던 것 같다.  
> 근데 그 새로움이 모래성의 시초가 될수도 있단 생각이 들어버렸다...

## 실행 계획
> 위 내용을 바탕으로, research와 plan에 대한 .md 파일은 코드 root에 docs 폴더로 관리할 필요가 있다고 판단.

### 네이밍 규칙
- spec-*: AI에게 요청하는 지침/규칙
- research-{yymmdd}-{hhmm}: 의사결정 이력(Decision Trail)을 위한 시간 이력 표시.
- plan-{yymmdd}-{hhmm}: research와 상동.

### 디렉토리 구조
```markdown
root/
├── docs/
│   ├── rules/                # 지침 파일 (spec-*.md)
│   │   ├── spec-workflow.md
│   │   ├── spec-research.md
│   │   ├── spec-plan.md
│   │   └── spec-implementation.md
│   ├── research/             # AI 생성 리서치 결과
│   │   └── research-260315-1538.md
│   └── plan/                 # AI 생성 및 사용자 검토 플랜
│       └── plan-260315-1540.md
└── ... (src, tests 등)
```

### 파일 단위 상세

`**docs/rules/spec-workflow.md**`  
```markdown
# 지침: 바이브 코딩 마스터 워크플로우 (Master Workflow)

당신은 사용자의 요구사항을 분석하고 구현하는 AI 에이전트입니다. 반드시 아래 정의된 단계별 워크플로우를 준수하여 업무를 수행하십시오.

## 전체 프로세스 단계
1. **[리서치 단계]**: 요구사항 관련 코드베이스를 깊이 읽고 분석합니다. (`spec-research.md` 참조)
2. **[계획 단계]**: 리서치를 기반으로 상세 구현 계획을 세웁니다. (`spec-plan.md` 참조)
3. **[사용자 검토]**: 사용자가 계획서에 남긴 주석/메모를 반영하여 계획을 최종 확정합니다.
4. **[구현 및 마무리]**: 확정된 계획을 기계적으로 구현하고 이력을 남깁니다. (`spec-implementation.md` 참조)

## 공통 규칙
- **디렉토리**: 모든 결과물은 `docs/research/` 또는 `docs/plan/`에 저장합니다.
- **파일명 규칙**: `{step}-YYMMDD-HHMM.md` (예: `plan-260315-1538.md`)
- **단계 이동**: 사용자의 명시적인 승인이 있거나 지침에서 정의한 조건이 충족될 때만 다음 단계로 이동합니다.

## 시작 방법
사용자가 기능을 요청하면, 즉시 `docs/rules/spec-research.md`를 읽고 리서치 단계부터 시작하십시오.
```

`**docs/rules/spec-research.md**`
```markdown
# 지침: 코드 리서치 (Research Phase)

모든 작업의 시작은 기존 코드베이스에 대한 완벽한 이해에서 출발합니다.

## 수행 지침
1. **파일 생성**: `docs/research/research-YYMMDD-HHMM.md` 파일을 생성하십시오.
2. **심층 분석**: 사용자가 지정한 범위와 관련된 모든 파일을 읽으십시오. 겉핥기식이 아닌, 함수 내부 로직과 의존성까지 상세히 파악해야 합니다.
3. **분석 내용**:
    - 현재 구현 방식 및 아키텍처
    - 수정이 필요한 지점 및 관련 있는 다른 모듈
    - **Required**: 반드시 지켜야 할 기술적 제약
    - **Optional**: AI가 제안하는 개선/선택 사항
4. **주의 사항**: 절대 코드베이스를 벗어난 추측을 하지 마십시오.

## 완료 조건
`research-*.md` 작성이 완료되면 사용자에게 보고하고, "리서치 결과를 바탕으로 계획 단계를 시작할까요?"라고 확인받으십시오.
```

`**docs/rules/spec-plan.md**`
```markdown
# 지침: 구현 계획 수립 (Planning Phase)

리서치 결과를 코드로 옮기기 전, 전략을 세우는 가장 중요한 단계입니다.

## 수행 지침
1. **파일 생성**: `docs/plan/plan-YYMMDD-HHMM.md` 파일을 생성하십시오.
2. **필수 포함 항목**:
    - **FrontMatter**: `task`, `date`, `research_doc`, `status: "Draft"`를 포함하십시오.
    - **접근 방식**: 기술적 해결 전략 및 트레이드오프 설명.
    - **수정 파일**: 변경될 파일들의 정확한 경로 목록.
    - **작업 체크리스트**: 구현 단계에서 하나씩 체크할 상세 Todo List.
3. **구현 금지**: 이 단계에서는 어떤 코드 파일도 실제로 수정해서는 안 됩니다.

## 사용자 피드백 루프
- 사용자가 `plan-*.md` 파일 내에 인라인 주석이나 메모를 남기면, 해당 내용을 반영하여 문서를 업데이트하십시오.
- 사용자가 "이 계획대로 구현해줘"라고 승인할 때까지 반복합니다.
```

`**docs/rules/spec-implementation.md**`
```markdown
# 지침: 구현 및 최종화 (Implementation & Finalization)

승인된 계획을 실행하고, 프로젝트에 영구적인 의사결정 이력을 남깁니다.

## 1. 구현 실행
- `docs/plan/plan-*.md`에 작성된 체크리스트 순서대로 코드를 작성하십시오.
- 창의적인 판단보다는 계획된 내용을 정확히 실행하는 데 집중하십시오.

## 2. 최종 승인 및 Git 커밋
사용자가 최종 구현을 확인하고 커밋을 지시하면 다음을 수행합니다.
1. **커밋 메시지**: `type(scope): summary` 형식을 지키고, 하단에 관련 `plan.md`와 `research.md` 경로를 포함하십시오.
2. **해시 추출**: 생성된 커밋 해시를 복사하십시오.

## 3. 계획 문서 최종 업데이트
참조한 `docs/plan/plan-*.md` 파일을 다음과 같이 수정하여 마무리하십시오.

- **FrontMatter 업데이트**:
  - `status`: "Completed"
  - `commit_hash`: "실제 커밋 해시"
- **체크리스트 업데이트**:
  - 완료 항목: `- [x] 작업명 (Completed at commit {hash})`
  - 미수정 항목: `- [ ] 작업명 (Reason: 사유 기술)`

## 완료 보고
모든 업데이트가 끝나면 사용자에게 전체 작업이 완료되었음을 알리십시오.
```
